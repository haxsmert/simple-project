# Relay 活约定清单(收尾自查必跑)

> 为什么有这个文件:对话里的强调会随上下文衰减,**写成测试的约定零衰减、留在共识里的约定全被后续改动踩破过**(实锤:等高、移动端范式、跨层传参)。所以——每条约定要么写成测试,要么进这份清单在每轮收尾时重放。范式照搬 contest-v2 的 `ui-layout-checklist.md`。

## 一、模型与流程(多数已固化为测试,列出供对账)

- 主干四阶段 `state` ⊥ 挂起 `hold`(平行字段,挂起=原地举手不搬站);除完成外任何阶段可中断、可多次。
- 守卫七条:计划是推进门票 / 确认可跳计划不可 / 自批闸 / 原地改派保角色 / claim 只领无主 / decision 只归答复管 / 父完成前子必须全完成。**全部在后端机制层**,三种交互方式(Web/HTTP/MCP)同闸。
- 概念极简:角色四种(提问是动作不是角色)、边两种(depends_on/clarifies)、字段名实相符(plan_md=计划)。**新概念/新字段先过"说得清+有消费方"再进模型**。

## 二、界面诚实与呈现(每轮 UI 改动收尾重放)

1. 看着能点的必须真能点;该能点的不能是死的;引用=标题+编码+跳转,编码不许独自出场。
2. 动作要交东西,东西就有输入口;动作依据什么,依据渲染在动作旁;不递不属于你的动作(关卡在别人手里时如实说)。
3. 界面呈现"人的意图"不是"系统的模型";合法≠该给;错误信息说人话(哪条守卫拦的、该走哪条路),**永不吐 FK/UNIQUE 黑话**。
4. 键盘/读屏那一半:焦点不掉 body、图标配 sr-only、可及名说清关系。
5. **同视图卡片等高**(桌面基线 132;纵向单列流不适用)——加任何卡面元素后必须重量三视图高度集合。
6. 位置即优先级:列内 rank→priority→id 落位,不用"高/中/低"文字。
7. 名字位用真名(人类默认 admin),"你"只用于第二人称文案。

## 三、移动端(≤640,范式来自 contest-v2 §八~十一)

- **单屏零横滑是架构级保证**(html/body overflow-x hidden + CDP 实测 scrollWidth≤innerWidth)。
- 窄屏退单列纵向流,不是横滑;触控目标 ≥44px;无 hover 依赖。

## 四、工程纪律(每轮收尾)

- **跨层传参:字段扩展时沿调用链走完**(组件→App 转发→api→server→service→repo),显式列举的转发点是静默丢失高危区——优先补传输层测试。
- 改动收尾自查**从本清单生成,不从 diff 生成**(新功能偏置=旧约定衰减之源)。
- 验证协议:双端测试全绿(汇总行要亲眼看到数字)+ PC(1440)与移动(390)各截图目视 + 关键流转 curl/浏览器实测;后端改动必须重启进程,纯前端只需 build。
- 迁移:幂等、不丢数据、guard 用 PRAGMA 判列不搜 SQL 文本;停服确认端口空了再动库文件。
- 演练/测试用独立库(RELAY_DB/RELAY_MIRROR),绝不碰 data/relay.db。
- 面向用户的中文全角标点(UI 文案/文档;代码注释与日志随工程惯例)。

## 五、语义诉求的接法

- 原则型/模型型诉求:先一句话复述"我理解的模型"再动手——**警惕用现有实现的最小改动去解读新诉求**(劣化器)。
- 新任务类别(如移动端、通知、多人)动手前:先检索用户其他项目里已验证的范式(contest-v2 等),不闭门造。
